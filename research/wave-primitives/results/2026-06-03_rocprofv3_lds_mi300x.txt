=== What this proves (MEASURED, not ISA-inferred) ===
1. DPP reductions issue EXACTLY ZERO LDS instructions (SQ_INSTS_LDS = 0,
   SQ_LDS_IDX_ACTIVE = 0). The "zero LDS traffic" claim is now a hardware
   counter reading 0, not an inference from disassembly.
2. The bpermute baseline issues 25.2M LDS instructions and 100.7M LDS-active
   cycles — the entire latency gap traces to LDS traffic.
3. VALU instruction count is nearly IDENTICAL between dpp_nobcast (37.76M) and
   bpermute (37.78M). Same arithmetic work; the only difference is LDS. This
   isolates the causal variable: the speedup is the elimination of LDS traffic,
   not fewer arithmetic ops.
4. hipCUB uses LDS too (4.2M insts) — 6x less than the portable baseline (hence
   faster than it) but nonzero (hence slower than full-DPP). hipCUB also issues
   MORE VALU (58.7M) than our full-DPP path (37.8M): we win on both LDS and VALU.

=== Causal chain, now end-to-end measured ===
__shfl_down -> ds_bpermute -> 25.2M LDS insts / 100.7M LDS-active cycles -> 978us
full DPP    -> row_shr+row_bcast -> 0 LDS insts / 0 LDS-active cycles    -> 261us
